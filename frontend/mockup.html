Plan 00:00:34

Analyzed the existing codebase. The project uses Servant for routing and PostgreSQL via Hasql. No existing auth module. Architecture:

  1. Credential storage — Argon2id hashing, separate user_credentials table
  2. Session layer — JWT access tokens (15min) + opaque refresh tokens in Redis (30 days)
  3. OAuth2 — Account linking via email match, PKCE for Google and GitHub
  4. Rate limiting — Token bucket per IP + per account, Redis-backed

Core auth module done. Login handler with rate limiting and Argon2id verification:

module PureClaw.Auth where import PureClaw.Auth.Session (createSession, SessionToken(..)) import PureClaw.Auth.RateLimit (checkRateLimit, RateLimitResult(..)) import Crypto.Argon2 (verifyEncoded) -- | Authenticate a user with email and password loginHandler :: LoginRequest -> AppM LoginResponse loginHandler req = do -- Check rate limit before any DB work rl <- checkRateLimit (req.clientIP) (req.email) case rl of RateLimited retryAfter -> throwError $ err429 { errBody = rateLimitBody retryAfter } Allowed -> pure () -- Look up credentials mcred <- findCredentialsByEmail (req.email) case mcred of Nothing -> throwError err401 Just cred -> do let valid = verifyEncoded (cred.passwordHash) (req.password) if valid then do session <- createSession (cred.userId) pure $ LoginSuccess session else throwError err401

Review 00:03:48

The login handler returns err401 for both "user not found" and "wrong password" — good, prevents email enumeration. But there's a timing side-channel: the Nothing branch returns immediately while Just runs Argon2id. An attacker can distinguish valid emails by measuring response time.

Two decisions needed:

  1. Add a dummy Argon2id verification in the Nothing branch to equalize timing? Adds ~200ms to failed lookups but closes the side-channel completely.
  2. The plan specifies JWT access tokens (15min). For a user-facing app, this means re-auth every 15 minutes if refresh fails. Increase to 1 hour, or keep 15min and prioritize the silent refresh endpoint?
